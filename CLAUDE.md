# CLAUDE.md (プロジェクトメモリ)

## 概要

本プロジェクトの根本方針は **「作るものをはっきりさせてから AI のコーディングを始める」** こと。特に **UI（操作画面）の仕様** をワイヤフレーム・状態・遷移・要素一覧・アクセシビリティ方針として明文化し終えるまで実装フェーズに入らない（後述「AIコーディング開始ゲート」）。

## ⛔ Claude への最重要指示（先頭で必ず読むこと）

**ドキュメントは絶対に一括生成しない。1ファイル作るごとに必ず停止して、ユーザーの明示的な承認を待つ。**

- 1回の応答で作成・編集してよいドキュメントは **1ファイルだけ**
- 「全部作って」「まとめて」と言われていない限り、複数ファイルを先回りで書き出さない
- ファイルを1つ書き終えたら **必ず作業を止めて承認を求める**（フォーマットは「開発プロセス」節参照）
- 「OK」「承認」「次へ」「進めて」等の明示的な合図があるまで次のファイルに着手しない
- 詳細は「開発プロセス」節の「最重要ルール」を参照

## ドキュメントの分類

### 1. 永続的ドキュメント（`docs/`）

アプリケーション全体の「何を作るか／どう作るか」を定義する恒久的なドキュメント。すべて **HTML 形式**で出力し、共通CSS（`docs/assets/design-system.css`）を相対パスでリンクする。情報密度（表・SVG）・視認性・ブラウザでの直接共有性を優先する。

**作成順序（UI を先に固める）：**

1. **product-requirements.html** — プロダクト要求定義書
   ビジョン／ターゲットユーザー／主要機能／成功定義／ビジネス・機能・非機能要件／ユーザーストーリー／受け入れ条件／**ユースケース図**（要求レベルの成果物としてここに置く）

2. **ui-specification.html** — UI仕様書（**作るものを画面レベルで固める核**）
   以下の節を必ず含める：
   - **0. デザイン原則 / トーン**
   - **1. 画面一覧**（画面ID `SCR-001-home` 形式／画面名／目的／URL・ルート対応）
   - **2. ユーザーフロー / 画面遷移図**（SVG）
   - **3. 各画面詳細**
     - 3.x.1 目的
     - 3.x.2 ワイヤフレーム（SVG/HTML、PC + モバイル）
     - 3.x.3 UI要素一覧（要素名／型／役割／バリデーション／必須・任意）
     - 3.x.4 状態定義（空／読込中／エラー／成功／レスポンシブ差異）
     - 3.x.5 操作とイベント（ボタン・リンクごとに表形式で）
     - 3.x.6 アクセシビリティ（**WCAG 2.1 AA を最低基準**）
   - **4. 共通コンポーネント**（ボタン／フォーム／モーダル／トースト／確認ダイアログ）
   - **5. デザイントークン**（色／タイポ／アイコン／スペーシング）
   - **6. 画面外の振る舞い**（通知・トースト・モーダル・確認ダイアログの発火）
   - **7. 用語整合**（`glossary.html` との対応表。UIラベルの揺れ防止）

3. **functional-design.html** — 機能設計書（内部設計に純化）
   アーキテクチャ／システム構成図／データモデル（ER図含む）／コンポーネント設計／状態管理／永続化／エラーハンドリング／API設計。**ワイヤフレーム・画面遷移図はここに置かない**。

4. **architecture.html** — 技術仕様書
   テクノロジースタック／開発ツール／技術的制約／パフォーマンス要件

5. **repository-structure.html** — リポジトリ構造定義書
   フォルダ・ファイル構成／ディレクトリの役割／ファイル配置ルール

6. **development-guidelines.html** — 開発ガイドライン
   コーディング規約／命名規則／スタイリング／テスト／Git規約

7. **glossary.html** — ユビキタス言語定義
   ドメイン・ビジネス・UI/UX 用語／英日対応表／コード上の命名規則。**UIラベルは `ui-specification.html` と用語整合を取る**。

- **assets/design-system.css** — 共通スタイルシート（全HTMLが参照）
- **assets/template.html** — 新規HTML作成時の雛形
- **assets/images/** — 画像（PNG / SVG）

### 2. 作業単位のドキュメント（`.steering/[YYYYMMDD]-[開発タイトル]/`）

特定の作業における「今回何をするか」を定義する一時的なステアリングファイル。作業ごとに新しいディレクトリを作成し、完了後は履歴として保持する。**requirements / design は HTML、tasklist のみ Markdown**（tasklist は差分追跡を優先）。

- **requirements.html** — 要求内容（変更・追加機能の説明／ユーザーストーリー／受け入れ条件／制約事項）
- **design.html** — 設計（実装アプローチ／変更コンポーネント／データ構造変更／影響範囲）+ **UI影響セクション（必須）**
  - **UI変更がある場合**: 変更画面リスト（画面ID付き）／差分ワイヤフレーム（変更前・変更後）／新規・変更 UI要素／状態定義の変更点／`ui-specification.html` 更新の要否
  - **UI変更がない場合**: 「**UI変更なし**」と明示記入（省略不可）
- **tasklist.md** — タスクリスト（実装タスク／進捗状況／完了条件）

**ディレクトリ命名規則**: `.steering/[YYYYMMDD]-[開発タイトル]/`
例: `.steering/20250115-add-tag-feature/`, `.steering/20250120-fix-filter-bug/`

## 開発プロセス

### 最重要ルール: 1ファイルずつ作成し、必ず停止して承認を待つ

> **Claude は「ドキュメントを一括生成」してはならない。1ファイル作るごとに作業を止め、ユーザーの明示的な承認を待ってから次のファイルに進む。**

#### 厳守事項（Claude の動作仕様）

1. **1メッセージ = 1ファイル**
   1回の応答で作成・編集してよい **ドキュメントファイルは1つだけ**。共通アセット（`design-system.css` / `template.html`）も1つずつ。複数ファイルを並行して作成・先回りして書き始めることは禁止。

2. **必ず停止して承認を求める**
   ファイルを1つ書き終えたら、Claude は **必ずその応答内で作業を停止**し、以下のフォーマットでユーザーに承認を求める：

   ```
   ✅ [ファイル名] を作成しました。
   👉 内容をご確認ください。
   - このまま次のファイル（[次のファイル名]）に進んでよいですか？
   - 修正したい箇所はありますか？
   承認（「OK」「進めて」「次へ」等）をいただくまで、次のファイルには着手しません。
   ```

3. **明示的承認のないファイルは作らない**
   ユーザーから「OK」「承認」「進めて」「次へ」等の明示的な合図がない限り、次のファイルの作成に着手しない。「全部作って」「まとめて進めて」と指示された場合のみ例外（ただしその旨を確認したうえで進める）。

4. **承認待ち中にできること / できないこと**
   - できる: 直前に作ったファイルの修正、ユーザーからの質問への回答
   - できない: 次のファイルの新規作成、複数ファイルの同時編集、先回りの設計議論で別ファイルに書き込む行為

5. **TodoWrite の必須利用**
   ドキュメント作成タスクは TodoWrite で **1ファイル = 1タスク** として分割し、`in_progress` は常に1つだけ。ファイル作成完了 → 承認受領 → `completed` に変更 → 次タスクを `in_progress` に、の順を守る。

各ステップは **1ファイル単位** に分解し、上記「最重要ルール」に従ってユーザー承認を都度得ながら進める。

## AIコーディング開始ゲート

> **Claude（および人間の実装者）は、以下のすべてが満たされるまで実装コードを編集・新規作成してはならない。**
> `docs/` および `.steering/` 配下のドキュメント編集はこの限りではない（未充足項目を埋めるためのドキュメント編集は推奨）。

### チェックリスト

**【共通・必須】**
- [ ] `docs/` 配下7ファイル（product-requirements / ui-specification / functional-design / architecture / repository-structure / development-guidelines / glossary）がすべてユーザー承認済み
- [ ] `docs/ui-specification.html` で以下が満たされている: 全画面のワイヤフレーム ／ UI要素一覧（表形式）／ 状態定義（空・読込中・エラー・成功・レスポンシブ）／ 画面遷移図・ユーザーフロー ／ アクセシビリティ方針
- [ ] 当該作業の `.steering/[YYYYMMDD]-*/` の requirements.html / design.html / tasklist.md がそれぞれユーザー承認済み

**【UI変更を含む作業の場合・追加必須】**
- [ ] `.steering/[YYYYMMDD]-*/design.html` の「UI影響セクション」に変更画面リスト・差分ワイヤフレーム・新規/変更UI要素・状態の変更が記載され承認済み
- [ ] 恒久仕様の変更を伴う場合、`docs/ui-specification.html` 側も更新済み

**【UI変更を含まない作業の場合】**
- [ ] `.steering/[YYYYMMDD]-*/design.html` の「UI影響セクション」に **「UI変更なし」と明示** 記入済み（省略不可）

**【新規用語が発生した場合・追加必須】**
- [ ] `docs/glossary.html` への用語追加が承認済み、必要なら `ui-specification.html` の「7. 用語整合」も更新済み

### ゲートを満たさない状態で実装着手を求められたときの挙動

Claude は実装着手を断り、**未充足項目を箇条書きで提示**してドキュメント編集に着手する。ユーザーが明示的に「ゲートをスキップして進める」と指示した場合に限り、リスクを明示したうえで実装に進む。

## 図表・ダイアグラムの記載ルール

### 記載場所
設計図は関連する HTML 内に直接埋め込む。独立した diagrams フォルダは作らない。

- **ワイヤフレーム** / **画面遷移図** / **ユーザーフロー** → `ui-specification.html`
- **ユースケース図** → `product-requirements.html`
- **ER図・データモデル図** → `functional-design.html`
- **システム構成図** → `functional-design.html` または `architecture.html`

### 記述形式

1. **SVGインライン埋め込み（推奨）** — Mermaid Live Editor 等で作図 → SVG出力 → 直接貼り付け。CDN依存なしでオフライン閲覧可。

   ```html
   <svg viewBox="0 0 200 100" xmlns="http://www.w3.org/2000/svg">
     <rect x="10" y="10" width="80" height="40" fill="#e0e7ff" stroke="#4338ca"/>
     <text x="50" y="35" text-anchor="middle">ユーザー</text>
   </svg>
   ```

2. **HTML + CSS** — 単純なフロー／ボックス図は `<div>` + Grid/Flexbox、表は `<table>`、折りたたみは `<details>`/`<summary>`。

3. **画像ファイル** — 複雑なモックアップは `docs/assets/images/` に PNG/SVG を配置し、相対パスで `<img>` 参照。

### 禁止・非推奨
- **Mermaid記法のCDN読込**：執筆時にSVGへ変換して埋め込む
- **ASCIIアート**：SVG/CSS で代替
- **重いJSフレームワーク**：単一HTMLファイルで完結すること

### 図表の更新
設計変更時は対応する図表も同時に更新。**UI仕様書のワイヤフレームと実装画面の乖離は最優先で解消**。

## 注意事項

### 画面ID / UI命名規則
- 画面IDは `SCR-[連番3桁]-[英小文字スラッグ]` 形式（例: `SCR-001-home`）
- 連番は `ui-specification.html` の「画面一覧」で採番し、欠番が出てもリナンバーしない
- 画面個別のワイヤフレームは原則 **`ui-specification.html` の章分け** で管理。画面数が大規模化（目安20画面超）した場合のみ `docs/ui/[screen-id].html` への分割を許容。

### HTML共通ルール
- ファイル間リンクはすべて **相対パス**（ローカルブラウザでもホスティングでも動くように）
- 情報密度優先、装飾過多を避ける
- ブラウザでファイルを開くだけで完結（追加サーバ・ビルドステップ不要）
- 共通テンプレート (`template.html`) をコピーして雛形を再利用
- コミットはファイル単位で意味のある単位に分ける
- コード変更後はリント・型チェックを必ず実施

## 共通テンプレートとCSSの仕様

### `docs/assets/template.html`
`<!DOCTYPE html>` ／ `<html lang="ja">` ／ `<meta charset="UTF-8">` ／ viewport meta ／ `<title>` プレースホルダ ／ `<link rel="stylesheet" href="assets/design-system.css">`（`.steering/` 配下からは `../../docs/assets/design-system.css` のように深さに応じて調整）／ 戻るリンク（インデックスへ）／ パンくず（章が多いドキュメント用）／ 目次プレースホルダ ／ `<main>` で本文を囲む

### `docs/assets/design-system.css`
- タイポグラフィ（日本語システムフォントスタック）／ 見出し h1〜h4・本文・リスト・リンクのベース
- `<table>` / `<code>` / `<pre>` / `<details>` のスタイル
- `<svg>` のレスポンシブ対応（`max-width: 100%`）
- レスポンシブ対応（最低 320px、メディアクエリで段組み）
- 印刷対応（`@media print`、A4想定、改ページ制御）
- ライト/ダーク両対応（`@media (prefers-color-scheme: dark)`）
- **UI仕様書向けクラス（推奨）**: `.wireframe`, `.ui-state-table`, `.screen-card`, `.token-swatch`
