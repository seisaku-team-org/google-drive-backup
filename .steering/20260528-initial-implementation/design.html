1. 実装アプローチの全体方針

プロダクト要求と機能設計はすでに確定しているため、本作業では 「仕様の通りに作る」ことを最優先する。 新規の意思決定はできるだけ避け、未決事項（機能設計書 §11）に該当する判断のみ本作業の中で行う。

1.1 守る前提

  • レイヤー依存方向（views → state → domain → infra）を最初から守る（後から直すのはコスト大）
  • infra 層のみが外部 IO を持つ。fetch や window.google を直接触るのは src/infra/ のみ
  • domain は React 非依存。CopyEngine / UrlParser / ReportBuilder は純粋関数（または小さなクラス）で書き、Vitest で単体テスト可能にする
  • State Store は useReducer + Context を 1 セットだけ。スライス（auth / job / ui）はファイル分割で表現

1.2 順序の原則

  1. 動かない基盤（依存・設定）を整える
  2. UI ガワだけ作って配線する（API はスタブ）
  3. infra 層を実装して本物の API に接続する
  4. CopyEngine を実装して再帰複製を動かす
  5. エラー・中止・レポートを仕上げる
  6. CI / デプロイを通す

各段階で動く状態を維持し、まとまった単位で PR を切る。

2. 段階別の実装計画

tasklist.md でこの段階をタスクに展開する。本書はその段階の中身を設計レベルで規定する。

フェーズ A: プロジェクト立ち上げ

  • Vite + React + TypeScript テンプレで npm create vite@latest 相当の最小プロジェクトを作る
  • tsconfig.json に strict: true / noUncheckedIndexedAccess: true を設定
  • ESLint / Prettier / EditorConfig を導入し、初回 lint を通す
  • Vitest + @testing-library/react のセットアップ、ダミーテストで green を確認
  • Playwright の最小セットアップ（実テストは後段）
  • src/styles/tokens.css に docs/assets/design-system.css のカラー・スペーシング変数をコピー
  • src/styles/global.css に body リセットとフォントスタックを設定
  • index.html に CSP メタタグ・GIS スクリプト・タイトルを設定
  • .gitignore / .editorconfig / README の最小版を整備

フェーズ B: 型とドメインの土台

  • src/types/index.ts に機能設計書 §3 の型を全部宣言（AuthState, UserInfo, FolderMetadata, CopyJob, CopyJobItem, Report, DriveItem）。FolderMetadata に isInSharedDrive / destinationParentId / canDuplicateHere、CopyJobItem.status に 'aborted' を含めること
  • src/domain/UrlParser.ts + テスト（5 URL 形式 + ID 単独 + 不正入力）
  • src/utils/backoff.ts + テスト
  • src/utils/semaphore.ts + テスト
  • src/utils/abortable.ts（任意）— AbortSignal を引数で受け取って待機やリトライをキャンセル可能にするヘルパー
  • src/domain/ReportBuilder.ts + テスト（集計とプレーンテキスト変換）。aborted カウントもサポート

フェーズ C: State Store と Router

  • src/state/authSlice.ts / jobSlice.ts / uiSlice.ts（Action + Reducer）
  • src/state/AppContext.tsx（Provider + 2 つの Context + hooks）
  • src/routes/router.tsx（HashRouter で 4 ルート定義）
  • 各 View をプレースホルダ実装（中身は <h2>HomeView</h2> 程度）し、画面遷移が動くことを確認

フェーズ D: 共通コンポーネントと UI ガワ

  • src/components/Button/（Primary / Secondary / Tertiary / Danger）
  • src/components/Header/（ユーザー識別 + ログアウトリンク）
  • src/components/Toast/ + uiSlice 経由の発火
  • src/components/ConfirmDialog/ + uiSlice 経由の発火
  • src/components/ProgressBar/
  • src/components/Spinner/
  • 4 View の UI を実装。データはモックで配線（ボタン押下で alert ではなく、UI 仕様書 §3 の状態遷移を Story 的に確認できる）
  • HomeView では「情報を取得」ボタンを持たず、URL 入力の onChange に debounce(500ms) を掛けて自動取得をトリガする実装に固定する
  • HomeView のプレビュー状態として「複製可能」と「複製不可（親階層に書き込み権限なし）」の 2 種を実装、後者では「複製を開始」を disabled、エラー文言（UI 仕様書 §3.2.4）を表示
  • ProgressView の進捗バー幅は aria-valuenow と連動。total が処理中に増加した場合、バーが瞬間的に逆戻りしても表示しっぱなしにする（強制 monotonic にはしない）

フェーズ E: infra 層

  • src/infra/AuthClient.ts（GIS を document.head に注入し、initTokenClient をラップ）
  • src/infra/driveTypes.ts（Drive API レスポンスの型。capabilities / driveId を含める）
  • src/infra/DriveApiClient.ts（fetch + Bearer + 429/5xx リトライ + 並列セマフォ）
    • 全メソッドが options.signal?: AbortSignal を受け取り、内部 fetch に伝搬する
    • getFolder(id, { fields: "...,driveId,capabilities(canAddChildren,canCopy)" }) をデフォルト fields とする
    • findAvailableName(baseName, parentId, { signal }) を実装（機能設計書 §7.5、10 件バッチ + 11 以降は逐次照会）
    • リトライ中であっても signal.aborted なら即座に AbortError を上に投げる（バックオフ待ちを中断するため src/utils/abortable.ts を活用）
  • infra 層のテストは「fetch を vi.fn() で差し替えて期待リクエストを検証」する単位で
    • signal.abort() → 進行中 fetch が AbortError を投げる確認
    • findAvailableName で 0 件 / 1 件 / 10 件 / 11 件以上の各ケースを確認

フェーズ F: CopyEngine

  • src/domain/CopyEngine.ts（機能設計書 §7 の疑似コードを実装）。内部で 1 個の AbortController を保持、全 driveApi 呼び出しに同じ signal を渡す
  • 事前バリデーション 3 ステップを冒頭で実行：
    1. 元フォルダ取得 → driveId が返れば SHARED_DRIVE_UNSUPPORTED エラー
    2. parents[0] が "root" でなければ親フォルダ取得 → capabilities.canAddChildren が false なら NO_PARENT_WRITE_PERMISSION エラー
    3. findAvailableName で衝突しない複製先名を決定（"Copy of XXX" → "Copy of XXX (2)" …）
  • start メソッドは { cancel, done } を返す。cancel() は controller.abort() を呼ぶ
  • 列挙と複製を並行する BFS。onProgress の total は処理中に増加してよい
  • 個別エラーは catch → スキップ続行、AbortError のみは catch して該当アイテムを 'aborted' 記録した上で再 throw → ループ脱出
  • DriveApiClient インターフェースをモック注入できる形にし、Vitest で以下を検証：
    • 3 階層の小さな構造を再帰的に複製できる
    • 個別エラー（403 / 404）でスキップして続行する
    • cancel() 呼び出しで進行中の処理が即座に AbortError で失敗し、aborted で記録される
    • 共有ドライブのフォルダを指定したらエラーで開始失敗する
    • 親階層書込権限なしのフォルダを指定したらエラーで開始失敗する
    • findAvailableName が空き番号を返す（0 件 / 既存あり / (2)あり / (2)(3)あり の各ケース）
    • onProgress / onItemStarted / onItemFinished / onJobFinished が呼ばれる
    • onProgress の total が増えるケース（再帰中にサブフォルダを展開）を確認

フェーズ G: 全結合 + ハッピーパス確認

  • 各 View のモックを外し、State Store + CopyEngine + DriveApiClient + AuthClient を結合
  • 本物の OAuth クライアント ID を .env.local に設定し、手動でハッピーパスを確認（テスト用 Drive アカウント / 小さなフォルダ）
  • UI 仕様書 §3 の状態（空 / 読込中 / エラー / 成功）を全画面で目視確認
  • Playwright E2E（スタブモード）で 1 本のハッピーパスを書く

フェーズ H: CI / デプロイ

  • .github/workflows/ci.yml（PR: typecheck + lint + test）
  • .github/workflows/deploy.yml（main: build + GitHub Pages へ deploy）
  • vite.config.ts の base をリポジトリ名に設定
  • GitHub Secrets に本番用 VITE_GOOGLE_CLIENT_ID を登録
  • 初回デプロイを実行し、本番 URL からハッピーパスを確認

フェーズ I: 仕上げ

  • README に「セットアップ手順」「OAuth クライアント ID の取得方法」「npm スクリプト一覧」を記述
  • 本書中の未決事項（後述 §8）の決定結果を、永続ドキュメントに反映
  • レポートのプレーンテキストコピーが動作することを確認
  • アクセシビリティの最低確認（キーボード操作 / focus 可視 / aria-live）

  • <label for> による入力欄ラベリング
  • エラーメッセージの aria-live="polite"
  • 進捗バーの role="progressbar" + aria-valuenow
  • キーボード操作（Tab / Enter / Esc）の動作確認
  • フォーカスリングを :focus-visible で維持
  • カラーコントラスト 4.5:1（design-system.css のトークン使用で担保）

6.7 用語整合（glossary との整合）

実装上のコード命名は 用語集 の英語 (code) 列に従う：

  • 複製先 = destination（dest / target 禁止）
  • 複製元 = source（src / original 禁止）
  • スキップ = skipped（個別アイテムでは failed としない）
  • 中止 = cancel（ユーザー操作の総称）、内部 fetch 中断は abort / AbortError / AbortSignal を使用
  • 個別アイテムが中止対象になった状態 = aborted（'skipped' とは区別）

6.8 ui-specification.html 更新の要否

本作業の事前時点では更新不要。実装の過程で仕様との不整合が見つかった場合のみ、同 PR で仕様を更新する（更新時は本 design.html も追記する）。

8. 本作業中に決定する未決事項

機能設計書 §11.2「未決」のうち、本作業中に結論を出すもの。決定したら永続ドキュメントに反映する。

※ 2026-05-28 アップデート: 「OAuth スコープの最終決定」は本ステアリング作成後に drive.readonly + drive.file 確定（Picker 不採用）として決着済み。 当該項目は本リストから除外し、永続ドキュメント側に確定として反映済み。

  1. 並列度の確定

    暫定 4 並列。フェーズ G で 10〜30 アイテムのテストフォルダを使い実測し、 429 発生頻度を観察。必要に応じ 2〜6 の範囲で調整し、 機能設計 §7.2 に確定値を記入する。

  2. 長時間処理時のトークン更新方針

    GIS の requestAccessToken 自動再取得が効くかをフェーズ G で実機検証。 効けば現状の実装で十分。効かなければ「処理中断 → 再ログイン依頼」のフォールバックで MVP は妥協する。

  3. findAvailableName の照会上限

    機能設計書 §7.5 では 10 件バッチ + 11〜99 を逐次照会としたが、実利用では 2 〜 3 件あれば十分なはず。 フェーズ G で挙動確認後、上限を調整する（例: 99 → 20 に下げる）。
