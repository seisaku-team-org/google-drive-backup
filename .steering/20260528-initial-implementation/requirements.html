1. 作業の目的

プロダクト要求定義書 §3.1 に定義された MVP（最初に作る最小版）を、 ブラウザで動作する Web アプリとして実装する。 「とにかく最小：1 フォルダを一回コピーできる」を最優先する。

本作業の完了時には、ユーザーが本アプリを開いて Google ログインを行い、 Drive フォルダの共有 URL を貼って「複製を開始」を押すと、 同じ親階層に 「Copy of 元フォルダ名」として丸ごとコピーされ、 完了時に成功・スキップのレポートが表示される ─ そこまでが動く。

FR-02ログイン継続時の自動進入本作業では実装しない（毎回ログインさせる。トークン期限内のみメモリで継続）

3. 非スコープ（本作業では実装しない）

プロダクト要求 §3.2「将来拡張」および §12「非スコープ」のものは、本作業では対象外とする。具体的には：

  • 双方向同期・差分同期・自動定期実行
  • 複数フォルダの一括複製・キュー処理
  • Google Picker API によるフォルダ選択 UI
  • 世代管理（自動で日付付き複製を定期作成）
  • 共有設定・コメント・版履歴の追従コピー
  • 共有ドライブへの対応
  • ローカル PC への書き出し・ダウンロード
  • モバイルブラウザ専用最適化
  • レポートの CSV 出力
  • ログイン継続（FR-02 の自動再ログイン）
  • i18n / 多言語化
  • ダークモード専用デザイン（CSS の prefers-color-scheme に従う範囲のみ）

4. 対象ユーザーストーリー

プロダクト要求 §8 のユーザーストーリー US-01〜05 のうち、本作業ですべて満たす：

US-01 — フォルダの保険コピーを作りたい

個人 Drive ユーザーとして、重要な資料が入ったフォルダを大きく編集する前に、同じ階層に「Copy of …」として完全な複製を作っておきたい。

US-02 — 雛形フォルダを新案件用に流用したい

フリーランスとして、案件雛形フォルダを、新しい案件のたびに複製して名前を付け替えて使いたい。

US-03 — 提出前のスナップショットを残したい

学生として、論文フォルダを提出する直前に「Copy of 卒論_2026春」として残しておきたい。

US-04 — 大量ファイルでも安心して進めたい

小規模事業者として、数百ファイル入った業務フォルダを複製するとき、途中で失敗しても全体は止まらず、最後にどれが失敗したか分かる状態にしたい。

US-05 — 進捗を見ながら待ちたい

すべてのユーザーとして、複製が走っている間、「今どのファイルを処理中か」「あと何件か」が見えていてほしい。

名前変更（US-02 で「新しい案件のたびに 名前を付け替えて使いたい」と書かれている部分）については、複製先名は「Copy of 〇〇」固定で作成されるため、複製後ユーザーが Drive 上で名前変更する想定（本アプリ内では行わない）。

5. 受け入れ条件

プロダクト要求 §9 の AC-01〜05 を本作業の完了基準とする。以下に再掲し、本作業に追加する条件を末尾に並べる。

AC-01: ログイン

  • 「Google でログイン」ボタンを押すと、Google の認証画面に遷移する
  • 認証後、本アプリのメイン画面（SCR-002-home）に戻り、ヘッダーにログイン中のメールアドレスが表示される
  • 「ログアウト」操作で SCR-001-login に戻る
  • OAuth キャンセル時はトーストでフィードバックされ、ボタンが再度活性に戻る

AC-02: フォルダ指定とプレビュー

  • 有効な Drive フォルダの共有 URL を貼り付けると、フォルダ名・所在パス・アイテム数のプレビューが表示される
  • UI 仕様書 §3.2.4 のエラー文言 5 種（URL 解析失敗 / フォルダではない / 404 / 403 / 共有ドライブ内）が状況に応じて表示される
  • 機能設計書 §4.2 UrlParser で定義した URL 5 形式 + ID 単独入力をすべて受け付ける

AC-03: 複製の実行

  • 「複製を開始」を押すと、元フォルダと同じ親階層に「Copy of 〇〇」フォルダが作成される
  • 元フォルダ内のすべてのサブフォルダ・ファイルが、階層構造を保ったまま複製される
  • Google Docs / Sheets / Slides もそのまま編集可能な状態で複製される
  • 元フォルダ・元ファイルには一切の変更が加えられない（読み取りのみ）

AC-04: 進捗とエラー

  • 処理中、進捗率（X% (n/total)）と現在処理中のアイテム名が画面に表示され続ける
  • 個別ファイルでエラーが発生しても全体は止まらず、次のアイテムに進む
  • 完了時にレポート画面が出て、成功件数・スキップ件数・スキップ詳細（名前・理由・所在）が一覧できる
  • レポート画面から複製先フォルダを Drive で直接開くリンクがある
  • 「中止」操作は確認ダイアログを経て、現在処理中のアイテム完了後に停止する

AC-05: 安全性

  • 本アプリは元フォルダ・元ファイルに対して書き込み・削除・移動を行わない（コードレビューおよび手動確認で担保）
  • アクセストークンが localStorage / sessionStorage / Cookie に書き出されない（コードレビューで担保）
  • OAuth で要求するスコープは drive.file + drive.readonly + openid email profile に限定する

AC-06（本作業追加）: 開発インフラ

  • プロジェクトが npm install → npm run dev でローカル起動できる
  • npm run typecheck / npm run lint / npm run test がそれぞれ green でパス
  • npm run build が成功し、dist/ が生成される
  • main マージで GitHub Actions が GitHub Pages へ自動デプロイし、本番 URL からアクセスできる

AC-07（本作業追加）: ドキュメント整合

  • 実装の結果として、永続ドキュメントに記述と異なる挙動が出た場合は、同 PR で永続ドキュメント側を修正する
  • 未決事項（機能設計書 §11）について判断したものは、永続ドキュメントに反映する

6. 前提・制約

6.1 前提

  • 開発者は Google Cloud Console で OAuth クライアント ID を発行できる
  • テスト用 Google アカウントと、十分小さなテスト用フォルダ（10〜30 アイテム規模）を用意できる
  • GitHub リポジトリ・GitHub Pages の設定権限がある
  • Node.js 20 LTS と npm 10 がローカルで動く

6.2 制約

  • 本番 Drive API へのアクセス上限: 1 ユーザー 100 秒あたり 1,000 リクエスト。指数バックオフ（機能設計書 §8.2）で守る
  • アクセストークン有効期限 60 分: MVP では長時間処理の途中失効は許容範囲とし、AC-01 の挙動（401 検知で再ログインへ）でフォールバック
  • テスト時の API クォータ消費: E2E は本番 API ではなくスタブで実施するため、開発中の API 呼び出しは個別動作確認時のみ
  • OAuth スコープ未決の課題: 機能設計書 §11「未決事項 1」（drive.readonly が強権限）について、本作業中に再検討してプロジェクトの結論を出すこと。本作業の初期実装は readonly + drive.file の併用案で進める
