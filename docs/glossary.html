0. 本ドキュメントの読み方

本用語集は本プロジェクト内のすべての文書とコードで用いる正式な呼称を定義する。次の原則に従う：

  • UI ラベルとして用いる場合は「UIラベル」列、コード上の英語表記としては「英語 (code)」列を用いる
  • 「避ける表現」に挙げた語を意図せず使ってしまったらレビューで指摘し直す
  • 新しい用語が登場したら必ず本書に追記する（development-guidelines.html §6 参照）
  • UI 上の表現については UI 仕様書 §7「用語整合」 も併読する

1. Google Drive 関連用語

正式名称UIラベル英語 (code)定義避ける表現
フォルダ フォルダ folder Drive 上の application/vnd.google-apps.folder mimeType を持つアイテム。中に他のアイテムを含むことができる 「ディレクトリ」「ホルダー」
ファイル ファイル file フォルダ以外の Drive アイテム全般（通常ファイル / Google 独自ファイル / ショートカット） 「ドキュメント」（範囲が狭い）
アイテム アイテム item / DriveItem 「フォルダまたはファイル」の総称。Drive API の files リソースに対応 「データ」「もの」「リソース」
Google 独自ファイル —（UI ラベルとしては基本使わない） googleNativeFile Google Docs / Sheets / Slides / Forms / Drawings 等、Drive 上でだけ編集可能な独自形式のファイル。mimeType が application/vnd.google-apps.* で始まる（folder を除く） 「Googleドキュメント」（Docs 限定の意味になる）
ショートカット — shortcut mimeType application/vnd.google-apps.shortcut のアイテム。実体は別のアイテムを参照するエイリアス 「リンク」「エイリアス」
マイドライブ マイドライブ myDrive ユーザー個人の Drive のルート領域。本アプリは MVP ではマイドライブ内のフォルダのみを対象とする 「個人ドライブ」
共有ドライブ 共有ドライブ（非対応の旨を伝える文脈でのみ使用） sharedDrive 組織で共有される Drive 領域。MVP では対応しない 「チームドライブ」（旧称）
ファイル ID —（内部表現） fileId / folderId Drive アイテムの一意な識別子。共有 URL のパスから抽出する 「ID」（単独だと曖昧）
共有リンク URL 共有 URL（短縮形） shareUrl Drive アイテムを共有するための URL。本アプリではこの URL を入力としてフォルダを特定する 「リンク」「アドレス」（汎用すぎる）
親フォルダ — parent / parents あるアイテムを直接含むフォルダ。Drive 仕様では複数親を許容するが本アプリは先頭の親を使う 「上位フォルダ」
アクセス権 アクセス権 permission あるユーザーがアイテムに対して持つ権限（owner / writer / reader） 「権限」（曖昧）

2. 複製処理に関する用語

正式名称UIラベル英語 (code)定義避ける表現
フォルダ複製 複製を開始 duplicate / copyFolder 本アプリの中核機能。元フォルダと同じ親階層に「Copy of 元フォルダ名」を作成し、中身を再帰的にコピーする一連の処理 「コピーする」「ダウンロードする」「実行する」
複製元 / 複製元フォルダ —（必要なら「元のフォルダ」） source / sourceFolderId ユーザーが複製したい元のフォルダ。本アプリは元データに対して読み取りのみを行う 「src」「original」「元データ」
複製先 / 複製先フォルダ 複製先フォルダ destination / destinationFolderId 本アプリが新規作成するフォルダ。命名は「Copy of 元フォルダ名」固定 「dest」「target」「保存先」「コピー先」
"Copy of" プレフィックス —（自動付与なので UI から明示） COPY_OF_PREFIX 複製先フォルダ名の先頭に付与する固定文字列。Google Drive 標準のファイル複製の命名規則と合わせている 「コピー -」「複製 -」
ジョブ —（内部用語） CopyJob 1 回の複製処理を表す単位。元フォルダ ID・複製先フォルダ ID・ステータス等を保持する 「タスク」「処理」
ジョブアイテム — CopyJobItem ジョブの中で複製対象となる個別のアイテム単位の状態 —
スキップ スキップ skipped / skipReason 個別アイテムの複製失敗時に、その 1 件だけ飛ばして全体処理は続行する挙動。「失敗」ではなく「スキップ」と呼ぶ 「失敗」「エラー」（個別アイテムについて）
中止 中止 cancel / cancelRequested ユーザーが複製処理を能動的に止めること。現在処理中のアイテム完了後に停止する 「キャンセル」（こちらはフォーム入力のクリア用）／「停止」「強制終了」
キャンセル キャンセル clear / resetInput SCR-002-home で URL 入力欄とプレビューをクリアする操作。複製処理の中止とは別概念 —
リトライ —（内部） retry API レート制限や一時的なサーバエラー（429 / 5xx）で同じリクエストを再試行する処理 「再試行」（コード内では statement の重複を避けるため英語）
指数バックオフ — exponentialBackoff リトライ間隔を 2 倍ずつ広げていく待機戦略（初回 1秒 → 2秒 → 4秒 …、上限 30秒） —
進捗 進捗 progress ジョブ全体の処理状況（n / total、X%） 「プログレス」
完了レポート レポート Report ジョブ完了時に表示する成功・スキップ件数のサマリーと、スキップ詳細を含む情報 「結果」「サマリー」
所要時間 所要時間 durationMs ジョブの開始から完了 / 中止までに要した時間。MM:SS 形式で表示 「処理時間」「経過時間」
ジョブフェーズ —（内部） JobPhase / 'counting' | 'copying' 1 回の複製処理を 2 段階に分けたフェーズ。counting（集計）→ copying（複製）と遷移する。2026-05-29 設計変更で導入 「ステージ」「ステップ」（曖昧）
集計フェーズ 集計中… counting 複製を始める前に、元フォルダ配下の全アイテムを BFS で列挙して総数を確定する段階。UI では「集計中…」見出しとスピナー＋件数カウンターを表示し、進捗バーは出さない 「カウント」「列挙」（曖昧）
複製フェーズ 複製中… copying 集計済みのアイテムを実際に Drive 上で複製する段階。total が固定されているので進捗バーは正確に 0%→100% に進む 「コピー」（用語整合 §7 参照）
列挙キャッシュ —（内部） childrenCache 集計フェーズで列挙した folderId → DriveFileResource[] マップ。複製フェーズで再利用することで listAllChildren を 1 フォルダ 1 回に抑える —

3. 認証に関する用語

正式名称UIラベル英語 (code)定義避ける表現
ログイン Google でログイン / ログアウト signIn / signOut OAuth2 で本アプリに Drive API へのアクセス許可を与える操作 / 解除する操作 「サインイン」「認証する」「ログオン」
セッション セッション（「セッションが切れました」等） session ログイン状態が有効である期間 —
アクセストークン —（一般ユーザー向けには出さない） accessToken Google API 呼び出しのために用いるトークン文字列。本アプリではメモリのみに保持 「トークン」（短縮しすぎ）
OAuth クライアント ID — VITE_GOOGLE_CLIENT_ID Google Cloud Console で発行する、本アプリを識別する OAuth 用の ID。シークレットは含まない 「API キー」
スコープ — scope OAuth でユーザーから要求する権限の範囲を表す URL 形式の識別子（例: drive.readonly） 「権限」（曖昧）
Google Identity Services（GIS） — google.accounts Google が提供するブラウザ向け OAuth 認証ライブラリ。https://accounts.google.com/gsi/client を読み込んで使う 「Google Sign-In」（旧称）

4. UI / UX 用語

正式名称UIラベル英語 (code)定義避ける表現
画面 — view / screen ユーザーに対して 1 つのまとまった目的を提示する UI 単位。本アプリでは 4 画面（SCR-001〜004） 「ページ」（本アプリは SPA だがハッシュ単位の「画面」と呼ぶ）
画面 ID — SCR-NNN-name 各画面に付与される一意な識別子。形式は SCR-001-home —
ワイヤフレーム — — 各画面の構成要素・配置を最小限のスタイルで示した図。UI 仕様書に PC・モバイル両用版を必ず描く 「モックアップ」（より作り込まれたものを指す）
コンポーネント — Component React の関数コンポーネント、または UI 仕様書 §4 で定義する共通 UI 部品 「パーツ」「部品」（仕様文書としては避ける）
ビュー — View 「画面」のコード上での呼称。src/views/HomeView/ のように対応する —
プレビュー / プレビューカード プレビュー preview / FolderPreview SCR-002-home で取得したフォルダ情報（名前・所在・件数・複製先名）を提示する UI ブロック 「サマリー」「情報パネル」
進捗バー — ProgressBar 0〜100% を視覚的に示す UI 要素（<progress> または role="progressbar"） 「プログレスバー」（カタカナ表記）
カウンター — Counter 成功・スキップ・残り件数を大きく数値表示するカード型 UI 「カウント」「数値」
トースト —（UI には出ない、内部呼称） Toast 画面の角に短時間表示される非ブロッキングな通知。表示時間は 4 秒が基準 「通知バー」「スナックバー」
確認ダイアログ —（モーダル内の見出しで「複製を中止しますか？」等） ConfirmDialog 「中止」「ログアウト中」など破壊的・引き返しにくい操作の前にユーザー確認を取るモーダル 「ダイアログ」（単体だと曖昧）
モーダル — Modal 背景を覆って前面に表示される UI。確認ダイアログを実装する基底 「ポップアップ」
デザイントークン — CSS カスタムプロパティ（--color-* 等） 色・タイポ・スペーシング等の値を再利用可能な変数として定義したもの 「変数」「テーマカラー」（曖昧）

5. アーキテクチャ用語

正式名称英語 (code)定義
SPA（シングルページアプリケーション） SPA 1 つの HTML から JS で各画面を描画する Web アプリ形態。本アプリの基本形態
静的ホスティング static hosting HTML / JS / CSS を配信するだけのホスティング。サーバロジックを持たない。GitHub Pages を採用
ハッシュルーター hash router URL の #/... 部分でクライアント側ルーティングを行う方式。サーバ側リライト不要
レイヤー layer 本アプリのコード階層区分（views / state / domain / infra）。依存方向はこの順に右向きのみ許可
State Store store / AppContext アプリ全体の状態を保持する仕組み。React の useReducer + Context で実装
スライス slice State Store の一部領域（auth / job / ui）。Redux Toolkit の呼称を借用
Copy Engine CopyEngine フォルダ複製の中核ロジック。再帰トラバース・進捗イベント・中止判定を担う
URL Parser UrlParser 共有 URL からフォルダ ID を抽出する純粋関数モジュール
Drive API Client DriveApiClient Google Drive API への HTTP リクエストを薄くラップするモジュール。リトライと並列度制御を内蔵
Auth Client AuthClient GIS をラップした認証モジュール
セマフォ semaphore 並列実行数を制限する同期プリミティブ。Drive API への同時リクエストを 4 に抑える

6. プロセス / 開発フロー用語

正式名称定義
永続的ドキュメント docs/ 配下に置かれる恒久的な仕様文書。アプリの「正」
作業単位のドキュメント / ステアリング .steering/YYYYMMDD-開発タイトル/ 配下に置く一時的な作業文書（requirements / design / tasklist）
UI 影響セクション ステアリングの design.html に必須で含める節。UI 変更の有無と詳細、または「UI 変更なし」を明示する
AI コーディング開始ゲート 実装着手前にすべての必須ドキュメントが揃っていることを確認するチェックリスト。CLAUDE.md「AIコーディング開始ゲート」節参照
受け入れ条件（Acceptance Criteria） 機能が「完了」と判定されるための具体的な条件群
ユーザーストーリー ペルソナの立場で「〜したい、なぜなら〜だから」と書いた要求の単位
ユースケース システムとアクター間の相互作用を抽象化した単位。図はユースケース図として表現
KPI（成功定義） プロダクトの成功を定量的に判定するための指標群。プロダクト要求 §4 に定義
ハッピーパス エラーが起きずに最後まで進む基本フロー。E2E テストの第一の対象
