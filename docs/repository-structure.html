2. トップレベルディレクトリの役割

パス役割編集者・編集タイミング
.github/workflows/ GitHub Actions の CI / デプロイ定義 CI 改修時のみ。原則 PR レビュー必須
.steering/ 作業単位の一時ドキュメント（YYYYMMDD-開発タイトル/ ごと） 新しい作業を始めるたびに新規ディレクトリ作成。CLAUDE.md「機能追加・修正時の手順」参照
docs/ 恒久的な仕様ドキュメント。アプリの「正」 UI 仕様 / 機能設計 / 用語などの変更時。実装と同時にコミット
public/ Vite が無加工でビルド成果に含める静的ファイル（favicon 等） ビルド対象外のアセット追加時のみ
src/ アプリ本体ソースコード 機能実装のメイン
tests/ ユニット・E2E テスト 機能実装と同期
index.html Vite が読むエントリー HTML。<meta>・CSP・GIS スクリプト読み込みを記述 初期設定後はほぼ触らない
CLAUDE.md 本リポジトリでの AI 共同作業のメタルール ルール改訂時のみ
README.md 「これは何のリポジトリか」「セットアップ手順」「主要 npm スクリプト」 初期セットアップ後・主要変更時

3. src/ の内訳

機能設計書 §4「コンポーネント設計」のレイヤー区分に対応する。レイヤー間の依存方向は views → state / domain → infra とし、逆流は禁止。

3.1 ディレクトリと責務

ディレクトリ役割典型ファイル
src/views/ 画面コンポーネント。UI 仕様書の SCR-001〜004 と 1:1 対応 LoginView.tsx, LoginView.module.css, LoginView.test.tsx
src/components/ 2 つ以上のビューで再利用する UI 部品 Header, Toast, ConfirmDialog, Spinner, ProgressBar, Button
src/state/ アプリ状態（Context + Reducer）。スライス単位でファイル分割 AppContext.tsx, authSlice.ts, jobSlice.ts, uiSlice.ts
src/domain/ ビジネスロジック。React 非依存。純粋関数を志向 CopyEngine.ts, UrlParser.ts, ReportBuilder.ts
src/infra/ 外部 IO（OAuth / Drive API / fetch）。本層のみが fetch や window.google を直接触る AuthClient.ts, DriveApiClient.ts, driveTypes.ts
src/types/ ドメイン共通型（複数レイヤーで使う型のみ） index.ts（AuthState, CopyJob, CopyJobItem, FolderMetadata 等）
src/styles/ グローバル CSS とデザイントークン tokens.css, global.css
src/utils/ レイヤーをまたぐ小さな汎用関数 backoff.ts, semaphore.ts
src/routes/ React Router の設定 router.tsx

3.2 1 コンポーネント = 1 フォルダの慣例

views / components 配下では、コンポーネントごとにフォルダを切り、3 ファイル構成を基本とする：

HomeView/
├── HomeView.tsx              # 実装
├── HomeView.module.css       # スコープ付きスタイル
└── HomeView.test.tsx         # コンポーネントテスト

サブコンポーネントが必要なら同フォルダ内に _PreviewCard.tsx のように先頭アンダースコアで private 扱いを示す。外部から import する想定があるなら別フォルダに昇格する。

3.3 依存方向ルール（重要）

FromTo許可
viewsstate / components / types / utils / stylesOK
statedomain / types / utilsOK
domaininfra（インターフェース経由） / types / utilsOK
infratypes / utilsOK
domain / infraviews / stateNG
domainfetch 直接呼び出しNG（infra 経由）

違反は ESLint の import/no-restricted-paths ルールで自動検出する設定を入れる（development-guidelines.html 参照予定）。

4. tests/ の内訳

4.1 tests/unit/（Vitest）

  • 対象: src/domain/ / src/utils/ の純ロジック、src/views/ のコンポーネント描画
  • 命名: {対象モジュール名}.test.ts(x)
  • 配置: テスト対象と同階層（views 配下のコンポーネントテスト）にするか、tests/unit/ に集約するかをモジュールに応じて選ぶ：
    • View / Component → 同階層（例: src/views/HomeView/HomeView.test.tsx）
    • 純ロジック → tests/unit/ に集約（例: tests/unit/UrlParser.test.ts）
  • モック: DriveApiClient 等のインターフェースを Vitest の vi.fn() で差し替える

4.2 tests/e2e/（Playwright）

  • 対象: ハッピーパス（ログイン省略のスタブ環境で URL 入力 → 複製 → レポート確認）
  • スタブ: fixtures/stub-server.ts で OAuth と Drive API をローカルでフェイク。本番 API には接続しない
  • 実行: npm run test:e2e。CI では PR には含めず、main マージ時のみ実行（実行時間と安定性のバランス）

5. 設定ファイル一覧

ファイル役割主な設定値
package.json 依存とスクリプト "type": "module" / npm スクリプトは 技術仕様書 §3.1 参照
tsconfig.json TypeScript 設定（アプリ用） strict: true, noUncheckedIndexedAccess: true, jsx: "react-jsx", target: "ES2022"
tsconfig.node.json Node 環境用（vite.config.ts などをチェック） include: ["vite.config.ts", "vitest.config.ts"]
vite.config.ts Vite 設定 base: "/<repo>/"（GitHub Pages 用）, React プラグイン
vitest.config.ts Vitest 設定 environment: "jsdom", setupFiles: ["./tests/setup.ts"]
playwright.config.ts Playwright 設定 webServer: { command: "npm run dev" }, baseURL は localhost
.eslintrc.cjs ESLint 設定 React / Hooks / TS / Prettier 互換、import/no-restricted-paths でレイヤー逆流禁止
.prettierrc 整形ルール singleQuote / semi / printWidth: 100 / trailingComma: "all"
.editorconfig エディタ共通設定 LF / UTF-8 / インデント 2 スペース
.gitignore — node_modules/, dist/, .env.local, Playwright の test-results/

7. 命名規則

対象規則例
ディレクトリ（コンポーネント） PascalCase HomeView/, ConfirmDialog/
ディレクトリ（カテゴリ） kebab-case 単数 views/, state/, infra/
React コンポーネントファイル PascalCase + .tsx HomeView.tsx
ロジック / ユーティリティ camelCase + .ts（クラス的なら PascalCase + .ts） backoff.ts, CopyEngine.ts
CSS Module 同名 + .module.css HomeView.module.css
テストファイル 同名 + .test.ts(x) HomeView.test.tsx, UrlParser.test.ts
E2E スペック kebab-case + .spec.ts happy-path.spec.ts
private サブコンポーネント 先頭アンダースコア _PreviewCard.tsx
型（ドメインエンティティ） PascalCase（type または interface） CopyJob, FolderMetadata
定数 UPPER_SNAKE_CASE MAX_RETRIES, BACKOFF_INITIAL_MS
環境変数 VITE_ プレフィックス + UPPER_SNAKE_CASE VITE_GOOGLE_CLIENT_ID
ステアリングディレクトリ YYYYMMDD-開発タイトル（小文字英 + ハイフン） 20260528-initial-implementation

画面 ID（SCR-001-home）と用語（複製先フォルダ 等）の命名整合は UI 仕様書 §7 および 用語集 を正とする。
