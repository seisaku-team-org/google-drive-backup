1. コーディング規約

1.1 TypeScript

  • strict モード必須。tsconfig.json の strict: true / noUncheckedIndexedAccess: true を変更しない
  • any 禁止。型が分からない外部値は unknown で受けてから型ガードで絞る
  • as キャストは最終手段。!（non-null assertion）も同様。使う場合は近くにコメントで根拠を残す
  • 型は type を基本。interface は拡張・宣言マージが必要なときだけ使う
  • 列挙は const 由来のユニオン型を優先（enum は使わない）： 
    // Good
export type JobStatus = 'idle' | 'preparing' | 'running' | 'cancelling' | 'completed' | 'cancelled' | 'failed';

// 避ける
export enum JobStatus { Idle, Preparing, /* ... */ }
  • 型定義は repository-structure.html §3 のルールに従い、複数層で使う型のみ src/types/ に置く

1.2 React コンポーネント

  • 関数コンポーネントのみ。class コンポーネントは使わない
  • Props は type Props = { ... } で型付け。React.FC は使わない（暗黙の children が混じるため）
  • 1 ファイル 1 公開コンポーネントを原則とする。同ファイル内の補助コンポーネントは先頭アンダースコアで private を示す
  • JSX 内のロジックは控えめに。条件分岐が 2 段を超えそうなら関数に切り出す
  • 例： 
    // HomeView.tsx
import styles from './HomeView.module.css';

type Props = {
  onStart: (folderId: string) => void;
};

export function HomeView({ onStart }: Props) {
  // hooks は先頭にまとめる
  const [url, setUrl] = useState('');
  // ...
  return (
    <main className={styles.root}>
      <h2>複製したいフォルダを指定してください</h2>
      {/* ... */}
    </main>
  );
}

1.3 hooks の使い方

  • hooks のルール（呼び出し順固定 / 条件付き呼び出し禁止）を厳守。eslint-plugin-react-hooks の警告は無視しない
  • カスタムフックは use プレフィックス。例: useCopyJob(), useAuth()
  • useEffect の乱用を避ける。「描画後の同期処理」「外部システムへの subscribe」など本来の用途以外には使わない。Props の派生値は計算式で表現する
  • 非同期処理は useEffect 内で async IIFE を使い、unmount 時のキャンセル可能性を考慮する

1.4 状態管理（useReducer + Context）

// src/state/AppContext.tsx
const AppStateContext = createContext<RootState | null>(null);
const AppDispatchContext = createContext<Dispatch<Action> | null>(null);

export function AppProvider({ children }: { children: ReactNode }) {
  const [state, dispatch] = useReducer(rootReducer, initialState);
  return (
    <AppStateContext.Provider value={state}>
      <AppDispatchContext.Provider value={dispatch}>
        {children}
      </AppDispatchContext.Provider>
    </AppStateContext.Provider>
  );
}

export function useAppState() {
  const ctx = useContext(AppStateContext);
  if (!ctx) throw new Error('useAppState must be used within AppProvider');
  return ctx;
}

export function useAppDispatch() {
  const ctx = useContext(AppDispatchContext);
  if (!ctx) throw new Error('useAppDispatch must be used within AppProvider');
  return ctx;
}
  • State と Dispatch は別の Context に分ける（dispatch しか使わない子コンポーネントの不要な再描画を防ぐ）
  • Action は discriminated union で定義する： 
    type Action =
  | { type: 'auth/signInSucceeded'; payload: { token: string; user: UserInfo } }
  | { type: 'auth/signOut' }
  | { type: 'job/start'; payload: { sourceFolderId: string } }
  // ...

1.5 エラーハンドリング

  • 独自エラークラスは最小限。DriveApiError 1 種で HTTP ステータスと reason を保持し、呼び出し側がスイッチする
  • try/catch は最も近い境界で。CopyEngine の各アイテム処理単位で catch し、スキップ判定する
  • throw new Error('something') は禁止。最低でも throw new Error(\`copy failed: \${item.id}\`) のように識別情報を含める
  • ユーザーに見せる文言とログ用文言を分離。エラーオブジェクトに userMessage を持たせるか、変換層で別関数にする

1.6 副作用の置き場所

副作用置き場所
fetch / Drive API 呼び出しsrc/infra/DriveApiClient.ts のみ
OAuth フロー（GIS）src/infra/AuthClient.ts のみ
クリップボード書き込みsrc/utils/ または直接 view から
window.open（複製先リンクを別タブで）view 内で OK

3. スタイリング規約

3.1 CSS Modules の使い方

  • コンポーネント単位でスコープ化。HomeView.module.css のクラスは styles.previewCard で参照
  • 1 コンポーネント = 1 module ファイル。グローバル CSS は src/styles/global.css と tokens.css のみ
  • クラス名は camelCase。CSS Modules では JS 側で styles.foo-bar が書きづらいため

3.2 デザイントークンの使い方

  • 色・スペーシング・角丸はすべて CSS カスタムプロパティ経由で参照する（直接の HEX 値・px 値は書かない）
  • トークン定義は src/styles/tokens.css に置き、docs/assets/design-system.css と整合させる
  • 例： 
    /* HomeView.module.css */
.previewCard {
  background: var(--color-surface);
  border: 1px solid var(--color-border);
  border-radius: var(--radius);
  padding: 0.75rem 1rem;
}

.actionButton {
  background: var(--color-accent);
  color: #fff;
}

3.3 レスポンシブ

  • モバイルファースト。デフォルトはモバイルレイアウト、@media (min-width: 768px) で PC レイアウト
  • ブレークポイント: 768px（タブレット以上 = PC レイアウト）。それ以上の細かい刻みは MVP では使わない

3.4 アクセシビリティに直結するスタイル

  • フォーカスリングを消さない。:focus-visible でアウトラインを明示
  • テキストコントラスト 4.5:1 以上を --color-text / --color-text-muted / --color-bg の組み合わせで担保。新しい色を導入する場合は事前にコントラスト計算する
  • アニメーションは prefers-reduced-motion を尊重

3.5 やってはいけないこと

  • !important の濫用（どうしても必要な場合はコメントで理由を残す）
  • インラインスタイル（style={{ ... }}）の常用。動的に変わるサイズ・位置などに限定する
  • グローバルセレクタ（body, h1 等）への上書きを CSS Modules 内で行う（global.css 経由のみ）

4. テスト規約

4.1 何をテストするか

対象テスト方針レイヤー
UrlParser 5 形式以上の入力 / 不正入力 / ID 単独入力をカバー Unit（Vitest）
CopyEngine DriveApiClient をモックして、再帰トラバース / 並列度 / cancel / 個別エラースキップ を検証 Unit
ReportBuilder 成功・スキップ件数の集計、所要時間、プレーンテキスト変換 Unit
各 View 状態（空 / 読込中 / エラー / 成功）ごとの描画 ／ 操作イベントが正しい dispatch を投げる Component（Vitest + Testing Library）
ハッピーパス全体 ログイン省略のスタブ環境で、URL 入力 → プレビュー → 複製 → レポートの 1 本道 E2E（Playwright）
backoff / semaphore 境界値（最大リトライ達成 / 並列度上限）と happy path Unit

4.2 何をテストしないか

  • Drive API そのもの（Google が責任を持つ）— 本番 API へは CI からアクセスしない
  • React / React Router の動作（フレームワーク自体のテスト）
  • CSS の見た目（ビジュアル回帰テストは MVP 対象外）
  • カバレッジ 100% を目指さない。重要パスを優先する

4.3 書き方

  • AAA パターン（Arrange / Act / Assert）でブロックを分ける
  • 1 テスト 1 アサーションを原則とする（複数の確認をしたい場合は describe で束ねて it を分ける）
  • テスト名は日本語可。「何が起きたら何になるか」を文として書く
  • 例： 
    describe('UrlParser', () => {
  it('共有 URL から folderId を抽出する', () => {
    const result = parseFolderUrl('https://drive.google.com/drive/folders/1AbC');
    expect(result).toEqual({ ok: true, folderId: '1AbC' });
  });

  it('フォルダではないリンクは NOT_A_FOLDER を返す', () => {
    const result = parseFolderUrl('https://drive.google.com/file/d/1AbC/view');
    expect(result).toEqual({ ok: false, reason: 'NOT_A_FOLDER' });
  });
});

4.4 モックの方針

  • infra 層のみモック対象。domain / view のテストでは DriveApiClient や AuthClient を vi.fn() で差し替える
  • モックは 各テスト内で定義。グローバルモックは避ける（テスト間の汚染を防ぐ）
  • fake server（Playwright の page.route）は E2E でのみ使用

4.5 実行

  • ローカル: npm run test（Vitest watch）／ npm run test:e2e
  • CI: PR で typecheck + lint + test を必須化、E2E は main マージ時のみ
  • 失敗したテストを .skip / .todo で放置しない。原因を直すか、削除する

5. Git 規約

5.1 ブランチ戦略

  • main: 常にデプロイ可能な状態。直接 push 禁止
  • feature ブランチ: feat/<短い説明> 形式。例: feat/url-parser, feat/copy-engine
  • fix ブランチ: fix/<短い説明> 形式。例: fix/folder-name-encoding
  • chore ブランチ: 設定・依存更新など。例: chore/upgrade-vite
  • ブランチ寿命は短く（数日以内）。長期化したら小さく分割する

5.2 コミットメッセージ（Conventional Commits）

形式: <type>(<scope>): <subject>

type用途例
feat新機能feat(copy-engine): 中止フラグを各ループ前にチェックする
fixバグ修正fix(url-parser): 末尾スラッシュ付き URL を許容する
refactor挙動を変えない整理refactor(state): authSlice を独立ファイルに分離
testテストの追加・修正test(copy-engine): cancel 時のスキップ件数を検証
docsドキュメントdocs(ui-spec): SCR-002-home の状態定義を追記
style整形のみ（挙動変更なし）style: prettier 実行
chore設定・依存更新chore: vite を 5.2 に更新
ciCI 設定ci: PR で typecheck を必須化
  • subject は日本語でよい。50 文字以内
  • 本文（body）が必要なら 1 行空けて記述。なぜそうしたかを書く（何をしたかは差分から分かる）
  • 1 コミット = 1 論理的変更。混ぜない

5.3 PR 規約

  • PR タイトル: コミットメッセージと同形式（<type>(<scope>): <subject>）
  • PR 本文には以下を含める：
    • 目的: 何のための変更か（リンク: .steering/<date>-<title>/）
    • 変更内容: 主要な変更点を箇条書き
    • 動作確認: 手動 / 自動の確認内容
    • 影響範囲: 他機能への影響
  • CI green 必須。typecheck / lint / unit test が失敗している PR はマージしない
  • 1 PR の差分は概ね 400 行以内を目安にする。それを超えるなら分割を検討
  • マージ方式は Squash and Merge（main の履歴を 1 PR 1 コミットに圧縮）

5.4 .gitignore に必ず含める

  • node_modules/
  • dist/
  • .env, .env.local, .env.*.local
  • test-results/（Playwright）
  • coverage/（Vitest）
  • .vscode/settings.json（個人設定のみ。共有設定は別途）
  • OS 固有: .DS_Store, Thumbs.db

5.5 機密情報の取り扱い

  • OAuth クライアント ID やシークレットを git に絶対コミットしない。GitHub Secrets と .env.local（gitignore 済み）で管理
  • 誤って push した場合は速やかに Google Cloud Console で当該認証情報を破棄し、新しいものを発行する

6. ドキュメント保守ルール

6.1 ドキュメントとコードの同時更新

仕様に影響する変更は 同じ PR でドキュメントも更新する。コード PR と docs PR を分けない（乖離を防ぐ）。

コード変更の種類同時更新が必要なドキュメント
新画面追加 / 既存画面の UI 変更docs/ui-specification.html
新しい状態 / アクションの追加docs/functional-design.html §5
新エンドポイント呼び出しdocs/functional-design.html §10
新規依存ライブラリdocs/architecture.html §8
新規ディレクトリ / 命名の変更docs/repository-structure.html
新規用語の発生docs/glossary.html + docs/ui-specification.html §7

6.2 ステアリングディレクトリの運用

  • 各作業の前に .steering/YYYYMMDD-開発タイトル/ を作成し、requirements.html / design.html / tasklist.md を順に承認を得て作成（CLAUDE.md「機能追加・修正時の手順」参照）
  • UI 影響セクション必須: design.html に「UI 変更あり: 詳細」または「UI 変更なし」を明示する
  • 作業完了後もディレクトリは削除しない（履歴として残す）

6.3 AI コーディング開始ゲート

CLAUDE.md の「AIコーディング開始ゲート」のチェックリストを満たすまで実装コードを編集しない。docs / steering の編集は許可される。

7. レビュー観点

PR レビュー時に確認するチェックリスト。

7.1 ドメイン整合

  • [ ] 変更が UI 仕様書 / 機能設計書と矛盾していないか
  • [ ] 用語集と異なる呼称を導入していないか
  • [ ] 「複製先 / 複製元」「スキップ / 中止」等を取り違えていないか

7.2 設計健全性

  • [ ] レイヤー依存方向に違反していないか（views → state → domain → infra）
  • [ ] fetch や window.google を infra 層以外で直接触っていないか
  • [ ] 純粋関数として書けるものに無用な副作用が混じっていないか

7.3 安全性

  • [ ] 元フォルダ・元ファイルへの書き込み / 削除操作が含まれていないか
  • [ ] OAuth スコープが不必要に拡張されていないか
  • [ ] トークンが localStorage / Cookie に書き出されていないか

7.4 アクセシビリティ

  • [ ] フォーカスが操作可能か（キーボードのみで使えるか）
  • [ ] エラーメッセージが aria-live に出ているか
  • [ ] 色だけで意味を伝えていないか（アイコン・文言の併用）

7.5 テスト

  • [ ] 新規ロジックにユニットテストがついているか
  • [ ] エラー / スキップ系の分岐がカバーされているか
  • [ ] .skip / .todo が残っていないか

7.6 ドキュメント

  • [ ] 仕様に影響する変更は対応ドキュメントを同時更新したか
  • [ ] 新規用語があれば glossary に追加したか
