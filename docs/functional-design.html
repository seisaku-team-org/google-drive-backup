1. アーキテクチャ概観

本アプリは サーバを持たない純粋なクライアントサイド SPA として設計する。 ユーザーの Drive データはブラウザを経由して Google Drive API と直接やり取りし、自前サーバには一切保存・転送しない。

  • 方式: Static SPA（HTML/CSS/JS をホスティングするだけ、サーバロジックなし）
  • 認証: Google Identity Services（GIS）の OAuth2 implicit-style トークンフロー（PKCE 相当）でクライアント側にアクセストークンを取得
  • API 通信: ブラウザから直接 Google Drive API v3 を呼び出す
  • 状態保持: メモリ上のみ。リロードで状態リセット（プロダクト要求 NFR-04 / NFR-05）

この設計により、サーバ側にユーザーのファイル内容やトークンが流れる経路が物理的に存在しないため、プライバシー要件（BR-2 / NFR-05）を構造的に満たす。

1.1 採用しない方式とその理由

方式採用しない理由
サーバ側でトークン管理＋API中継 ファイル内容を経由させるとプライバシー要件と矛盾。サーバ運用コストも MVP には過剰。
ブラウザ拡張機能 NFR-06「追加インストール不要」と矛盾。
WebAssembly / Service Worker 常駐 MVP は「1回の複製」が完結すればよく、常駐は不要。
サーバサイドレンダリング SEO 不要、初回読み込みも軽量で SSR の動機が薄い。

4.2 主要モジュールの責務

CopyEngine

  • 引数: 複製元 FolderMetadata、DriveApiClient、進捗コールバック
  • 処理: 複製先フォルダ作成 → 子要素を BFS でトラバース → 各アイテムを複製 → 進捗イベントを通知
  • 状態: cancelRequested を見て次のアイテム処理前に停止判定
  • 出力: CopyJob と全 CopyJobItem の最終状態

UrlParser

  • 入力: ユーザーが貼り付けた文字列
  • 許容形式:
    • https://drive.google.com/drive/folders/{id}（一般的な共有 URL）
    • https://drive.google.com/drive/folders/{id}?usp=sharing
    • https://drive.google.com/open?id={id}
    • https://drive.google.com/drive/u/0/folders/{id}
    • ID 単独（30 文字前後の英数字 + アンダースコア / ハイフン）
  • 出力: { id: string } または { error: 'INVALID_URL' }

DriveApiClient

  • 共通: Bearer ヘッダー付与、JSON シリアライズ／パース、レスポンスステータスごとのエラー分類
  • メソッド: getFile / listChildren / createFolder / copyFile（次節「内部 API」参照）
  • リトライ: 429 / 5xx は指数バックオフ（1s → 2s → 4s → 8s → 16s、最大 5 回）
  • 並列制御: 同時 4 リクエストまで（セマフォ実装）

AuthClient

  • Google Identity Services（GIS）の google.accounts.oauth2.initTokenClient をラップ
  • メソッド: signIn() / signOut() / getToken() / onTokenExpired()
  • スコープ: https://www.googleapis.com/auth/drive.file + https://www.googleapis.com/auth/drive.readonly（後述「外部 API」節で詳細）

4.3 依存関係

循環依存は禁止。State Store は Domain・Auth を呼ぶが、Domain・Auth は State Store を直接参照しない（依存性逆転：必要な情報は引数で渡す）。

7. フォルダ複製アルゴリズム

7.1 全体フロー（疑似コード）

前提: CopyEngine は 1 つの AbortController を内部に持ち、すべての driveApi.* 呼び出しに同じ signal を渡す。 ユーザー操作「中止」で controller.abort() が呼ばれると、進行中の全 fetch が AbortError で失敗し、即座にループを抜ける。

2 フェーズ進捗（2026-05-29 設計確定）: ローカル動作確認で「分母が処理中に増える」UX が分かりにくいと判明したため、 Phase 1（集計）で先に総数を確定 → Phase 2（複製）で正確な進捗バーを表示する 2 段方式に変更した。 集計フェーズで列挙した children をキャッシュして複製フェーズで再利用するため、API 呼び出し総数は増えない。 

function startCopy(sourceFolderId):
  controller = new AbortController()
  signal = controller.signal
  childrenCache = new Map()   // folderId -> DriveFileResource[]

  // (1) 元フォルダのメタデータ取得（共有ドライブ判定を含む）
  src = driveApi.getFolder(sourceFolderId,
        fields = "id,name,parents,driveId,capabilities", signal)
  if src.driveId is set:
    throw 'SHARED_DRIVE_UNSUPPORTED'

  // (2) 複製先親階層の決定
  destinationParent = src.parents.length > 0 ? src.parents[0] : 'root'

  // (3) 親階層への書き込み権限を事前チェック（root 以外）
  if destinationParent != 'root':
    parent = driveApi.getFolder(destinationParent,
             fields = "id,capabilities", signal)
    if not parent.capabilities.canAddChildren:
      throw 'NO_PARENT_WRITE_PERMISSION'

  // (4) 連番付与で空いている名前を決定（§7.5 参照）
  destinationName = findAvailableName("Copy of " + src.name, destinationParent, signal)

  // ============================================================
  // Phase 1: 集計（BFS で全アイテム列挙 + キャッシュ + total 確定）
  // ============================================================
  progress.phase = 'counting'
  countQueue = [{ id: sourceFolderId, name: src.name }]
  while countQueue is not empty:
    if signal.aborted: throw AbortError
    current = countQueue.shift()
    progress.currentItemName = current.name
    emit(progress)

    children = driveApi.listAllChildren(current.id, signal)
    childrenCache.set(current.id, children)
    progress.total += children.length
    emit(progress)

    for child in children:
      if child is folder:
        countQueue.push({ id: child.id, name: child.name })

  // (5) 複製先フォルダ作成（集計後 — 中止時にゴミが残らない）
  dst = driveApi.createFolder(destinationName, destinationParent, signal)
  job.destinationFolderId = dst.id
  job.status = 'running'

  // ============================================================
  // Phase 2: 複製（キャッシュした children を使い、API は増えない）
  // ============================================================
  progress.phase = 'copying'
  queue = [{ srcId: sourceFolderId, dstId: dst.id, path: src.name }]

  while queue is not empty:
    if signal.aborted: break
    current = queue.shift()
    children = childrenCache.get(current.srcId)  // ★ キャッシュから取り出す

    // 同階層の子要素を並列で処理（Promise.allSettled、実効並列度は
    // DriveApiClient のセマフォで 4 に制限）
    results = await Promise.allSettled(children.map(child => processChild(child, current)))

    if any result is AbortError: break

  job.finishedAt = now()
  job.status = signal.aborted ? 'cancelled' : 'completed'
  emit('job.finished')

function processChild(child, parent):
  jobItem = recordItemStart(child, parent.path)
  try:
    if child is folder:
      newFolder = driveApi.createFolder(child.name, parent.dstId, signal)
      queue.push({ srcId: child.id, dstId: newFolder.id, path: parent.path + "/" + child.name })
    else:
      driveApi.copyFile(child.id, parent.dstId, signal)
    markSuccess(jobItem)
  catch err:
    if err.name == 'AbortError':
      markAborted(jobItem)
      throw
    markSkipped(jobItem, err.message)

function cancel():
  controller.abort()

7.2 並列度

単純逐次は遅いが、過剰な並列は 429 を誘発する。同階層内の子要素について、同時 4 件まで並列とする（セマフォで制御）。 フォルダ作成だけは順序が必要なため逐次（親 → 子）。

7.3 サブフォルダの扱い

  • サブフォルダは files.copy では複製できない。files.create で空フォルダを作り直し、中身を再帰的にコピーする
  • これにより元フォルダの ACL（共有設定）・コメント・版履歴は 引き継がれない（MVP ではこれを許容、要件 C-3）

7.4 ショートカット（application/vnd.google-apps.shortcut）の扱い

  • 子要素が mimeType === 'application/vnd.google-apps.shortcut' の場合、ショートカットアイテムとしてそのまま files.copy で複製
  • リンク先実体までは追従しない（要件 C-4）

7.5 同名複製先の扱い（連番付与）

「Copy of {元名}」と同じ名前のフォルダが同一親階層にすでに存在する場合、 「Copy of {元名} (2)」「Copy of {元名} (3)」… のように連番を自動付与して衝突を回避する。 Drive は同名併存自体は許容するが、ユーザー側で「どれが今回の複製か」を区別できなくなるため連番方式とする。

findAvailableName アルゴリズム

function findAvailableName(baseName, parentId, signal):
  // 1. baseName と "baseName (2)" 〜 "baseName (10)" を 1 回の files.list で検索
  query = `'\${parentId}' in parents and trashed=false and (
            name='\${baseName}' or
            name='\${baseName} (2)' or
            ... or
            name='\${baseName} (10)'
          )`
  existing = driveApi.listChildren(parentId,
             { q: query, fields: "files(name)", signal }).items
  existingNames = new Set(existing.map(f => f.name))

  if not existingNames.has(baseName):
    return baseName
  for n in 2..10:
    candidate = `\${baseName} (\${n})`
    if not existingNames.has(candidate):
      return candidate

  // 2. 10 を超える場合は (11) 以降を 1 件ずつ問い合わせ（実用上は稀）
  for n in 11..99:
    exists = driveApi.checkExists(parentId, `\${baseName} (\${n})`, signal)
    if not exists:
      return `\${baseName} (\${n})`

  throw 'TOO_MANY_DUPLICATES'   // 99 まで埋まっていることはまずない

事前に 10 件まではバッチで取得し、API 呼び出しを 1 回に抑える。それ以上は実用ケースとして稀なので逐次照会で十分。

7.6 中止時の挙動（AbortController による中断）

  • CopyEngine 内に 1 つの AbortController を保持。cancel() で controller.abort() を呼ぶ
  • すべての fetch（DriveApiClient 経由）に同じ signal を渡しているため、進行中のリクエストが即座に AbortError で失敗する
  • 各アイテムの処理で AbortError を識別したら、そのアイテムを aborted ステータスで記録し、例外を上位に伝播させてループを抜ける
  • 通常のエラー（403 / 404 / 5xx）は従来通りスキップして続行する。AbortError のみが中断扱い
  • すでに作成済みの複製先フォルダおよび複製済みファイルは 削除しない（要件 FR-14 準拠）
  • SCR-004-report で「中止」見出しと、それまでの成功件数 / 中止件数を表示する

進行中のリクエストを途中で中断するため、サーバ側で複製処理が一部完了している可能性がある（例: files.copy で複製が始まったがレスポンス前に中断された場合）。 その場合、複製先に「想定外の余分なファイル」が残ることがあるが、元データには影響しないため許容する。

8.2 指数バックオフ仕様

  • 初回待機: 1 秒 + ジッタ 0〜500ms
  • 2回目以降: 直前待機 × 2 + ジッタ。上限 30 秒
  • 最大リトライ回数: 5
  • HTTP 429 / 500 / 502 / 503 / 504 / ネットワーク例外でのみ発火

8.3 スキップ理由のフォーマット

レポート画面およびコピー用テキストでは、以下の形式で表示する。

{HTTPステータス}: {一行サマリー}
詳細: {Drive API エラーレスポンスの reason フィールド}
所在: {アイテムの相対パス}

10. 外部 API（Google Drive API）

10.1 OAuth スコープ

スコープ用途
https://www.googleapis.com/auth/drive.readonly 元フォルダのメタデータ取得・子要素一覧の取得（読み取り）
https://www.googleapis.com/auth/drive.file 本アプリが作成したファイル・フォルダへの書き込み
openid email profile ログイン中ユーザー情報（email / name）の取得

本構成は確定（2026-05-28）。drive.file は「本アプリが作成・開いたファイル」しかアクセスできないため、URL 貼り付け方式（ユーザー任意のフォルダ ID にアクセス）を支えるには drive.readonly の併用が必須。 Picker API への移行で drive.file 単独化する代替案は検討したが、URL 貼り付け方式の UX と両立しないため採用しないこととした（プロダクト要求 §3.2 参照）。

10.2 使用エンドポイント

用途メソッドエンドポイント主要パラメータ
フォルダメタデータ取得 GET /drive/v3/files/{folderId} fields=id,name,mimeType,parents,driveId,capabilities(canAddChildren,canCopy)
・driveId が返れば共有ドライブとして拒否
・親フォルダの capabilities.canAddChildren で書き込み権限を事前確認
子要素一覧取得 GET /drive/v3/files q='{parentId}' in parents and trashed=false
fields=files(id,name,mimeType,size),nextPageToken
pageSize=200
フォルダ作成 POST /drive/v3/files body: { name, mimeType: 'application/vnd.google-apps.folder', parents: [destinationParentId] }
ファイル複製 POST /drive/v3/files/{fileId}/copy body: { parents: [destinationParentId] }
fields=id,name
ユーザー情報取得 GET /oauth2/v3/userinfo —

10.3 共通リクエストヘッダー

Authorization: Bearer {accessToken}
Accept: application/json
Content-Type: application/json   // POST のみ

10.4 ページネーション

listChildren は nextPageToken が返らなくなるまで繰り返し呼ぶ。1 ページあたり 200 件を上限とする（API のデフォルト 100 より大きいが、Drive のドキュメント上限 1000 内）。

11. 未決事項 / 検討課題

11.1 確定済み

  1. OAuth スコープ（2026-05-28 確定）→ drive.readonly + drive.file + openid email profile で確定。Picker API への移行は採用しない（URL 貼り付け方式と両立しないため）。詳細 §10.1。
  2. 進捗 total の算出方法（2026-05-29 設計変更で確定）→ 2 フェーズ方式に変更。Phase 1（counting）で全アイテムを事前列挙して total を確定し、Phase 2（copying）で正確な進捗バーを表示。当初の「逐次更新」案は動作確認の結果「ユーザーが終わりを予測できない」と判明したため撤回。§7.1 参照。
  3. プレビューのアイテム数（2026-05-28 確定）→ 直下のみを表示（再帰総数は取らない）。
  4. 同名複製先の扱い（2026-05-28 確定）→ Copy of XXX (2)、(3) の連番付与。§7.5 に詳細。
  5. 中止時の挙動（2026-05-28 確定）→ AbortController で進行中の fetch も即座に中断。§7.6。
  6. マイドライブ直下フォルダの複製先（2026-05-28 確定）→ マイドライブ直下に作成（parents:["root"]）。
  7. 非オーナー編集可フォルダの対応（2026-05-28 確定）→ 元の親階層に書き込み権限がある場合のみ作成。書き込み権限がなければエラーで中止し、自動フォールバックはしない。
  8. 並列度（2026-05-29 動作確認後に確定）→ 同時 4 並列で確定。動作確認で 429 エラーは発生せず、レスポンスも安定していたため変更不要。
  9. 長時間処理中のトークン期限切れ（2026-05-29 動作確認後に確定）→ MVP では「401 検知 → 再ログインを促す」フォールバックで許容。動作確認で実行時間が短く期限切れシナリオに遭遇しなかったため、GIS 自動再取得の本格実装は将来拡張に回す。
  10. findAvailableName の上限（2026-05-29 動作確認後に確定）→ 10 件バッチ + 11〜99 逐次の現状実装で確定。動作確認では (2) までしか使われず、99 まで埋まる現実的なケースは無いと判断。
  11. テスト用モック（2026-05-28 確定）→ DriveApiClient インターフェースをファクトリパターンでモック差し替え可能にする方式で確定。src/services/stubServices.ts 参照。MSW は導入しない。

11.2 残課題（将来拡張）

  1. 巨大フォルダ対策: 数千件のアイテム数を持つフォルダでは、メモリ上に CopyJobItem 配列を全件持つと膨らむ。ストリーミング処理に変える余地あり。MVP のテスト対象外（要件 C-7）。実機計測で 1000 件超のシナリオは未確認なので、後日大規模フォルダで再評価する。
  2. 長時間処理時のトークン自動更新: 60 分超の処理を想定する場合は GIS の requestAccessToken でサイレント再取得を組み込む（現在は再ログイン誘導でフォールバック）。
