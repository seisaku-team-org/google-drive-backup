1. テクノロジースタック

1.1 ランタイム / 言語

レイヤー採用技術バージョン目標採用理由
言語 TypeScript 5.4+ Drive API レスポンスの型付け、CopyJob / CopyJobItem 等の状態モデルを型安全に扱うため必須
UI フレームワーク React 18.x エコシステムが厚く、Drive API 連携の前例も多い。4 ビュー規模ではフックベースで素直に書ける
ルーティング React Router（hash router） 6.x GitHub Pages 配信のため hash ルーター（#/login 等）。サーバ側リライト不要
状態管理 React の useState / useReducer + Context React 18 標準 4 画面・1 ジョブの規模で Redux 等は過剰。Context 1 つで auth / job / ui を提供
スタイリング 生 CSS + CSS Modules — ドキュメント側の design-system.css のトークンをそのまま継承。Tailwind 等の追加依存を入れない

1.2 ビルド / 開発サーバ

用途採用技術バージョン目標採用理由
ビルド / Dev サーバ Vite 5.x 起動・HMR が高速、React/TS テンプレートが公式に整備。GitHub Pages へのデプロイ設定も簡素
パッケージマネージャ npm 10.x 以上（Node.js LTS 同梱版） 追加インストール不要、lockfile が単一で扱いやすい
Node.js（開発時のみ） Node.js 20 LTS 以上（Node 24 LTS で動作確認済み） Vite / TS / Vitest / Playwright が動作する Node LTS。CI は Node 20 で実行、ローカルは winget が入れる現行 LTS（執筆時点 Node 24）で動作確認済み。package.json の engines.node は >=20

1.3 認証・API クライアント

用途採用技術備考
OAuth クライアント Google Identity Services（GIS） <script src="https://accounts.google.com/gsi/client"> を直接読み込み。npm 経由のラッパーは入れない
Drive API クライアント 自作（fetch ベース） googleapis（Node 専用）や gapi.client は使わず、ブラウザ標準 fetch を薄くラップ

機能設計書 §10 で定義した最小 5 エンドポイントだけを叩くため、専用 SDK を導入する利点が薄い。バンドルサイズ・依存ライセンス管理の観点でも自作が有利。

2. 開発ツール

2.1 静的解析・整形

用途ツール設定方針
型チェック TypeScript（tsc --noEmit） strict: true 必須。noUncheckedIndexedAccess も有効化
Lint ESLint @typescript-eslint + eslint-plugin-react + eslint-plugin-react-hooks。Prettier との衝突は eslint-config-prettier で解消
整形 Prettier シングルクォート、セミコロンあり、行幅 100、末尾カンマ all

2.2 テスト

レイヤーツール対象
ユニットテスト Vitest UrlParser / CopyEngine（DriveApiClient はモック注入）／ ReportBuilder 等の純ロジック
コンポーネントテスト Vitest + @testing-library/react 各 View の状態切替（空 / 読込中 / エラー / 成功）。Drive API は DriveApiClient インターフェースをモックで差し替え
E2E テスト Playwright ハッピーパス（ログイン省略のスタブ環境で）。本番 Drive API には接続せず、開発時起動の Vite サーバ + テスト用のスタブで動かす

Drive API のリアル接続テストは MVP 対象外（クォータ消費・テスト不安定化を避ける）。スモークテストとして手動で 1 度だけ実施し、レポートに記録する。

2.3 推奨 IDE / 拡張

  • VS Code
  • 拡張: ESLint / Prettier / EditorConfig / GitHub Actions（YAML 補完）
  • .editorconfig で改行コード LF・インデント 2 スペースを統一

  • 方式: GitHub Actions により main ブランチへのマージで自動デプロイ

  • 環境変数: Google OAuth クライアント ID は VITE_GOOGLE_CLIENT_ID としてビルド時に注入（リポジトリの GitHub Secrets で管理）

3.3 環境別の設定

環境OAuth クライアント ID許可リダイレクト URI

クライアント ID は Google Cloud Console の OAuth 2.0 設定で発行。アプリ側からはコードに直書きせず import.meta.env.VITE_GOOGLE_CLIENT_ID で参照する。

5.1 ロード性能（初回アクセス）

指標目標備考

JS バンドル合計（gzip）150 KB 以下React + Router 込み。CSS in JS や大型ライブラリの追加を避けることで達成可
Lighthouse Performance90 点以上（PC）—

5.2 実行時性能（複製処理）

シナリオ目標
100 アイテム規模のフォルダ（深さ 3 階層）体感 3 分以内（プロダクト要求 KPI と一致）
進捗 UI の更新間隔最大 1 秒（過剰更新は防ぐ）

5.3 メモリ

CopyJobItem の配列は 1 件あたり 200 バイト前後を想定。1,000 件で 200KB 程度。10,000 件規模は対象外（要件 C-7）。

5.4 API 利用量

呼び出し1複製あたりの回数
files.get1（プレビュー用）
files.listフォルダ数 × ページ数（200件/ページ）
files.create（フォルダ）サブフォルダ数 + 1
files.copyファイル数

合計呼び出し数の概算: 「ファイル数 + サブフォルダ数 × 2 + 1」程度。Google Drive の 1 ユーザー / 100 秒あたり 1,000 リクエストのレート上限内に通常収まる。

6. セキュリティ・プライバシー方針

6.1 OAuth スコープ

要求スコープは以下で確定（2026-05-28）：

  • https://www.googleapis.com/auth/drive.readonly — URL 貼り付け方式で任意フォルダのメタデータを読むために必須
  • https://www.googleapis.com/auth/drive.file — 本アプリが作成するフォルダ・ファイルへの書き込み
  • openid email profile — ログイン中ユーザーの識別表示

drive.readonly は Drive 全体読み取りに相当する強権限だが、URL 貼り付け方式（ユーザーが任意の Drive フォルダ ID を指定する）を採用する以上、これを併用する必要がある。 Picker API への移行で drive.file 単独化する代替案は検討したが、URL 貼り付け方式と両立しないため採用しないこととした（プロダクト要求 §3.2 / 機能設計書 §10.1 参照）。

書き込みは drive.file の範囲に閉じるため、本アプリは「ユーザーが Drive で別途編集しているファイル」を変更することは技術的にできない。元データへの破壊的操作は構造的に発生しえない。

6.2 トークン管理

  • アクセストークンは JS のクロージャ内変数として保持。localStorage / sessionStorage / Cookie には一切書き込まない
  • ページリロード時はトークンが失われ、再ログインを求める（実装が単純で、トークン漏洩リスクも最小）
  • トークン失効（401）を検知したら即座に未認証状態に戻す

6.3 プライバシー

  • サーバを持たないため、ユーザーのファイル内容・メタデータ・トークンが自前インフラを経由しない
  • 外形監視・アクセスログの類は MVP では設置しない
  • Google Analytics 等の第三者トラッキングは導入しない

6.4 CSP / 配信ヘッダー

GitHub Pages はカスタムヘッダー設定が限定的だが、<meta http-equiv="Content-Security-Policy"> で以下を最低限指定する：

default-src 'self';
script-src 'self' https://accounts.google.com https://apis.google.com;
connect-src 'self' https://www.googleapis.com https://oauth2.googleapis.com;
frame-src https://accounts.google.com;
img-src 'self' data:;
style-src 'self' 'unsafe-inline';

style-src に unsafe-inline を含めるのは、初期 HTML に小さなインラインスタイルを持たせるため。コンポーネントの CSS Modules は 'self' で済む。

6.5 依存ライブラリの監査

npm audit を CI で実行し、High 以上を検知したらブロック。Dependabot を有効化して PR で更新を回す。

7. 技術的制約

  1. Google Drive API のレートリミット: 1 ユーザー / 100 秒あたり 1,000 リクエスト、1 プロジェクト全体で 10,000 リクエスト / 100 秒。指数バックオフ実装で守る
  2. アクセストークンの有効期限: 通常 60 分。長時間処理では途中で切れる可能性がある
  3. フォルダの「コピー」API は存在しない: files.copy はファイル単体のみ。フォルダは「create + 再帰コピー」の合成で実装する（機能設計書 §7）
  4. GitHub Pages の制限: ビルド成果物 1GB まで、リクエスト 100GB/月、HTTPS 必須（OAuth リダイレクト URI 要件と整合）
  5. CSP 配信制約: GitHub Pages はレスポンスヘッダーをカスタムできないため、CSP は meta タグで指定。strict なルールは妥協が必要
  6. ブラウザの popup ブロッカー: OAuth ポップアップが弾かれることがあるため、ユーザー操作のクリック直後にしか開かない
